# TrustWeave Reference Wallet — iOS (Phase 2.5d scaffold)

Native iOS / SwiftUI wallet that mirrors the [Android Phase 2 wallet](../android/README.md)
and talks to the same in-repo demo backend.

## Status: UNVALIDATED SCAFFOLD

This Phase 2.5d delivery is **starter source code** that was written without Xcode
access. It compiles in concept but has not been built or run on a Mac. Treat it as
a head-start, not a working app — expect to fix a few Swift compile errors when
you first open it.

What's deliberately included:
- Full source layout mirroring the Android app (`Views/` + `Lib/`)
- CryptoKit Ed25519 (iOS 17+) for the holder signing key — `Curve25519.Signing.PrivateKey`
- Keychain (Security framework) for storing the holder identity + credentials
- URLSession client for the demo backend
- SD-JWT VC encode/decode/present (mirror of `android/shared/.../SdJwtVc.kt`)
- did:key encoding via hand-rolled Base58

What's deliberately excluded:
- No `.xcodeproj` / `.xcworkspace` — the file format is fragile and is best
  generated by Xcode itself.
- No iOS-side consumption of the KMM `shared` module yet. Phase 2.5e: configure
  the shared module's iOS targets, build it into an XCFramework, link it from
  this app, and remove the duplicated Swift `SdJwt.swift` + `Crypto.swift`.
- No Phase 2 polish features — minimal Home + Receive + Present screens only.

## Setup (when you have Xcode)

1. Open Xcode → File → New → Project → iOS App.
2. Product Name: `TrustWeaveWallet`. Interface: SwiftUI. Language: Swift.
   Target: iOS 17.0+ (CryptoKit Ed25519 requirement).
3. Drag the entire `TrustWeaveWallet/` directory into the new project's source tree.
   Xcode will offer to copy or reference — choose reference, with "Create groups".
4. In the new project's `Info.plist`, add `NSAppTransportSecurity` →
   `NSAllowsArbitraryLoads` = YES so the simulator can reach the local Next.js
   demo backend over HTTP. (Production removes this and requires HTTPS.)
5. Configure the backend URL. Edit `Lib/DemoBackend.swift` and change `baseURL`
   to your host machine's LAN IP, or `http://localhost:3000` for Mac-hosted
   simulator. (Unlike Android emulator's `10.0.2.2`, iOS simulator reaches
   localhost directly.)
6. Build & run on the simulator.

## File layout

```
ios/TrustWeaveWallet/
├── App/
│   └── TrustWeaveWalletApp.swift     SwiftUI App + TabView root
├── Views/
│   ├── HomeView.swift                Holder identity + credentials list
│   ├── ReceiveView.swift             Receive flow
│   └── PresentView.swift             Present flow + disclosure checkboxes
└── Lib/
    ├── Wallet.swift                  Facade: bootstrap/store/list/present
    ├── HolderKey.swift               Ed25519 keypair management (CryptoKit + Keychain)
    ├── Storage.swift                 Credential storage (Keychain for blobs,
    │                                 UserDefaults for the index)
    ├── DemoBackend.swift             URLSession client for issuer + verifier
    ├── Crypto.swift                  base64url + did:key + Base58 + JWS sign
    └── SdJwt.swift                   SD-JWT VC encode / decode / present
```

The structure mirrors Android. The `Lib/` files have the same conceptual surface as
`android/app/src/main/kotlin/.../lib/` — same method names, same flows.

## Known places that will need attention when you first build

1. **CryptoKit Ed25519 raw key conversion.** `Curve25519.Signing.PrivateKey` doesn't
   expose raw 32-byte form directly; the scaffold uses `.rawRepresentation` which
   should work but may need adjustment.
2. **Keychain access groups.** The Keychain code uses default access groups; if you
   later want shared keys (extensions, App Group), wire `kSecAttrAccessGroup`.
3. **SwiftUI imports.** Some files may need `import Combine` or `import Foundation`
   additions the unvalidated draft missed.
4. **URL string interpolation.** A couple of the `URL(string:)` callsites assume
   safe ASCII; if your DID happens to contain unusual characters, percent-escape it.

## Phase 2.5e: KMM consumption

The Android shared module at `../android/shared/` is ready to add iOS targets
(`iosX64`, `iosArm64`, `iosSimulatorArm64`). When you do that:

1. Add the iOS target blocks to `../android/shared/build.gradle.kts`.
2. Implement iOS actuals for `CryptoPrimitives.kt`:
   - `sha256` via `CC_SHA256` from CommonCrypto
   - `secureRandomBytes` via `SecRandomCopyBytes`
   - `ed25519Sign` / `ed25519Verify` / `ed25519GenerateKeyPair` via CryptoKit's
     `Curve25519.Signing.PrivateKey`
3. Build with `./gradlew :shared:linkPodReleaseFrameworkIosArm64` (or assemble
   XCFramework via `assembleSharedXCFramework` task).
4. Drop the framework into this iOS project; replace `Lib/SdJwt.swift` +
   `Lib/Crypto.swift` with `import Shared`.

After 2.5e, the iOS app's only SwiftUI-side code is the views and Keychain plumbing.
